# 深度调研

## 1. 范围界定

开始前，与用户确认（或从上下文推断）：

- **领域边界**：什么在范围内，什么明确排除？
- **深度**：概览、API 对比，还是源码级分析？
- **输出格式**：Markdown 报告、决策矩阵、带注释的文献目录，还是全部？
- **目标受众**：用于架构决策、文献综述，还是新人入门？
- **时间预算**：覆盖多少个框架/库/系统？深度调研默认 8–12 个。

## 2. 资源筛选

### 2.1 发现候选

用多种信号找到相关资源：

1. **社区共识**：GitHub stars、论文引用、顶会收录（NeurIPS、ICML、ICLR、OSDI、SOSP）
2. **生态位**：是基础层（如 PettingZoo、LLVM、PostgreSQL）还是应用层？
3. **维护健康度**：最近 commit、issue 响应速度、废弃项目的迁移指南
4. **领域重叠**：是否有目标领域的示例或案例研究？

### 2.2 优先级

| 优先级 | 标准 |
|---|---|
| P0 | 领域基础/标准/引用最多 |
| P1 | 设计哲学截然不同的主要替代方案 |
| P2 | 子问题的专用工具 |
| P3 | 领域专用适配（领域较窄时） |

排除条件：无人维护（>2 年）、无设计文档或架构说明、与已选 P0/P1 重复。

## 3. 并行探索

并发启动 4–6 个 explore agent，每个覆盖 1–2 个资源。

### Agent 提示模板

```
深入阅读 <资源>。关注：
1. 核心抽象：根本模型或不变量是什么？
2. 架构：关键模块、职责及组合方式
3. 入口：主要 API / CLI / 协议面
4. 状态管理：可变状态如何处理？（事务、epoch、episode）
5. 扩展点：用户如何定制或接入？
6. 性能模型：批处理、向量化、异步、JIT 等
7. 领域适配：是否有天然映射到目标领域的原语？

按此格式输出：
- 资源：<名称> <链接>
- 核心抽象：<一句话>
- 架构：<关键文件/模块列表>
- 入口：<列表>
- 状态管理：<机制>
- 扩展点：<扩展方式>
- 性能模型：<向量化/GPU/异步等>
- 领域适配评分：1–5
- 关键源码路径：<关键逻辑的文件路径与行号范围>
- 意外/陷阱：<非显然行为或坑>
```

### 协调规则

- 每个 agent 分配不重叠的资源
- 若资源有依赖（如 A 包装 B），分配给同一 agent 或确保顺序
- 每个 agent 设置超时（快速 15 分钟，深入 30 分钟）

## 4. 信息架构

结果返回时，组织到一份动态文档中：

### 4.1 分类

按设计哲学分组，而非按字母序。常用分类法：

- **按形式化模型**：如 MARL 的 POSG vs EFG vs AEC；协同编辑的 CRDT vs OT
- **按架构模式**：如集中式 vs 分布式；编译型 vs 解释型
- **按抽象层次**：如内核/基线 vs 中间件 vs 应用框架
- **按性能策略**：如向量化批处理 vs 事件驱动 vs JIT 编译

### 4.2 交叉参考矩阵

尽早构建矩阵，随 agent 返回持续更新。维度依领域而定，通常包括：

1. **核心抽象 / 形式化模型**
2. **API 风格**（声明式 / 命令式 / 函数式 / 消息传递）
3. **状态语义**（全局 / 局部 / 复制 / 事件溯源）
4. **可扩展性**（插件 / 钩子 / 继承 / 组合）
5. **性能策略**（批处理 / 向量化 / 异步 / JIT / GPU）
6. **正确性保证**（串行化 / 线性化 / 最终一致 / 确定性）
7. **运维复杂度**（单二进制 / 分布式 / 需要协调器 / 托管 vs 自建）
8. **领域示例**（开箱即用的典型用例）

## 5. 深度分析

### 5.1 对比分析

对同一问题的不同解法，逐对分析：

- **它们共同解决什么问题？** 精确陈述。
- **各自优先保证什么不变量？**（如一致 vs 可用、简洁 vs 性能）
- **权衡是什么？** 尽量用具体数字（延迟、吞吐、内存）。
- **何时选 A 不选 B？** 给出决策标准，而非观点。

### 5.2 源码级阅读

阅读源码时关注：

1. **基类 / 核心协议**：定义其他一切实现的契约
2. **状态转移**：`step()`、`apply()`、`execute()` 或 `process()` 方法
3. **生命周期钩子**：创建、初始化、销毁、错误处理
4. **边界层**：系统如何与外部交互（I/O、网络、用户）
5. **测试模式**：测试比文档更能揭示预期行为和边界情况

始终引用文件路径和行号范围。优先用永久链接（commit hash），避免用可变分支链接。

### 5.3 概念辨析

领域术语易混淆时，产出精确的辨析章节：

- 给出形式化定义（元组表示或等价形式）
- 展示层级 / 子集关系
- 列出常见错误并明确纠正
- 用调研来源中的实例来锚定抽象概念

## 6. 报告撰写

### 6.1 结构

```markdown
# <领域> 深度调研

## 1. 范围与方法
（调研了什么、为什么、怎么做的）

## 2. 核心概念
（形式化定义、分类法、概念辨析）

## 3. 来源分析
（每个来源一节，使用统一模板）

## 4. 对比
（定位矩阵 + API/设计对比表 + 特性矩阵）

## 5. 领域发现
（领域专用框架或适配方案）

## 6. 建议
（针对目标项目的具体、有据建议）
```

### 6.2 写作规范

- **公式**：行内用 `$...$`，行间用 `$$...$$`，`$$` 内单反斜杠。禁止用代码块包裹公式。
- **代码引用**：文件路径 + 行号范围。如 `src/env.py` L120–L150 (`step`)。API 名用行内代码，变量用行内公式。
- **链接**：源码引用用 commit hash 的永久链接，论文用 arXiv/DOI。
- **表格**：对比优先用 Markdown 表格，紧凑但可读。
- **断言**：每个论断都应可追溯至来源、代码路径或逻辑推导。

### 6.3 质量清单

交付前检查：

- [ ] 每个来源至少有一处引用的文件路径 + 行号范围
- [ ] 对比矩阵覆盖所有来源的所有关键维度
- [ ] 形式化定义与引用来源一致
- [ ] 无孤立论断（每个观点都有证据支撑，或明确标记为推断）
- [ ] 链接可点击且指向稳定 URL
- [ ] 公式渲染正确（`$$` 内无 `\{` 转义问题，无代码块包裹数学）

## 7. 迭代

用户阅读报告后，预期会有跟进问题：

- "为什么你把 X 排在 Y 前面？" → 指向对比矩阵 + 具体证据
- "能不能深入看 Z 的做法？" → 对 Z 的特定模块启动专注的 explore agent
- "这怎么应用到我的具体场景？" → 将建议映射到用户的约束条件

把首份报告当作基线，用跟进问题来深入钻取。
